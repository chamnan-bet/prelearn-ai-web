PLANNING: BFS, Dijkstra, A* for PreLearn.ai
===========================================


── BFS — "Related Patterns" panel ─────────────────────────────────────────

Problem it solves:
  When studying a pattern, show which other patterns are connected to it —
  so the student knows what else to review.

How it works in the project:
  - Build a graph where patterns are nodes and edges connect related patterns
    (same subject, shared prerequisite, or same exam year)
  - BFS from the current pattern outward — finds all patterns within 1–2 hops
  - Display them as "You should also study these"

Data needed:
  A edges map in mathPatterns.js listing which patterns connect to which.
  Example:
    p001 → [p002, p003, p004]   (all limits patterns)
    p004 → [p001, p029]          (L'Hôpital links to ODE characteristic equation)

UI:
  A small panel on PracticeView showing 2–3 related pattern chips.

Effort: Low — graph is small (30 nodes), BFS on 30 nodes is trivial.


── Dijkstra — "Optimal Study Path" ─────────────────────────────────────────

Problem it solves:
  Given your current mastery, find the most efficient ORDER to study all
  remaining patterns to maximize mark recovery.

How it works in the project:
  - Nodes = patterns, edges = study order relationships (prerequisites)
  - Edge weight = inverse of marksAtRisk (low weight = high priority)
  - Dijkstra finds the shortest weighted path from "current state" to "all mastered"
  - Output: an ordered list — "study p006, then p008, then p020..."

Data needed:
  Same prerequisite graph as BFS, plus numeric marksAtRisk weights
  (already parseable from the marks field in each pattern).

UI:
  A "My Study Path" page showing a numbered roadmap of patterns in optimal order.

Effort: Medium — need to define the full prerequisite graph carefully.


── A* — "Deadline-Aware Study Path" ────────────────────────────────────────

Problem it solves:
  Same as Dijkstra, but factors in the exam date — if only 5 days left,
  skip low-risk patterns and focus only on high-risk ones.

How it works in the project:
  - Dijkstra + a heuristic: h(pattern) = estimated marks still at risk
    if this pattern is skipped
  - The heuristic gets tighter as the exam date approaches — A* becomes
    more aggressive about prioritizing high-risk patterns
  - Output: same ordered path as Dijkstra but re-ranked based on days remaining

Data needed:
  Same graph as Dijkstra + exam date (user input or hardcoded).

UI:
  Same "Study Path" page but with a countdown and a note like:
  "3 days left — skipping medium-risk patterns"

Effort: High — depends on Dijkstra being built first; heuristic design needs thought.

2. Dijkstra    — the main showpiece feature, builds on BFS graph
3. A*          — only if time allows; Dijkstra already covers the core concept


── What BFS assists in this project ─────────────────────────────────────────

BFS assists the student in two specific ways:

1. Contextual awareness during practice
   When a student opens a pattern (e.g. p001 — "Not factoring a 0/0 limit"),
   BFS traverses the graph 1 hop outward and surfaces directly related patterns:
     p002 (rationalize), p003 (∞/∞), p004 (L'Hôpital)
   This tells the student: "These patterns share the same root skill — if you
   struggle here, you likely need these too."

2. Guided navigation between connected mistakes
   Each related pattern card is clickable. Clicking one navigates to that
   pattern's practice page, re-runs BFS from the new node, and shows a fresh
   set of neighbors. The student can explore the graph naturally without
   returning to the pattern list.

What BFS does NOT do (important distinction):
   - It does not rank patterns by urgency (that is Greedy's job)
   - It does not find an optimal study order (that is Dijkstra's job)
   - It only answers: "Given where I am, what is directly connected?"

The graph structure (src/data/patternGraph.js):
   - 30 nodes (patterns)
   - Edges based on: same subject group, shared prerequisite technique,
     or overlapping exam skill (e.g. p004 L'Hôpital ↔ p029 ODE char eq,
     because both require understanding derivatives)
   - BFS runs client-side in O(V+E) time — always instant, no API call needed

── Latest updates: making the project more genuinely AI ─────────────────────

Two upgrades were implemented to strengthen the AI credibility of the project.


UPGRADE 1 — Dynamic Greedy (data now comes from Firestore, not static files)
─────────────────────────────────────────────────────────────────────────────

Problem before:
  Greedy read from static mathPatterns.js. The weights never changed.
  Anyone could argue "this is just sorting hardcoded data, not AI."

What changed:
  1. src/services/patterns.js
     - seedPatterns() now writes masteredCount: 0 to each Firestore pattern doc
     - New incrementMasteredCount() uses Firestore atomic increment(1)

  2. src/composables/useProgress.js
     - markMastered() now calls incrementMasteredCount() after saving user progress
     - Every mastery event from any user bumps the global counter in Firestore

  3. src/composables/useGreedy.js
     - Added masteredCount as a tiebreaker in the sort
     - Primary: higher marksAtRisk → study first (same as before)
     - Tiebreaker: lower masteredCount → fewer users mastered it → still hard → higher priority
     - The recommendation now reflects real-world difficulty across all users

  4. src/views/HomeView.vue
     - Fetches live pattern data from Firestore on mount (includes masteredCount)
     - Greedy uses Firestore data; falls back to static mathPatterns.js while loading
     - Recommendation card is instant on first render, updates after Firestore loads

Why this matters:
  Greedy's weights are no longer hardcoded. They adapt as real students use the app.
  A pattern many users have mastered → lower priority. Still struggling → higher priority.
  This is a simple adaptive recommendation system driven by real user behaviour.

Action required:
  Re-run /dev/seed to add masteredCount: 0 to existing Firestore documents.
  Without this, incrementMasteredCount will fail on patterns missing the field.


UPGRADE 2 — Dijkstra: "Study Path" page (new AI algorithm)
───────────────────────────────────────────────────────────

Problem before:
  Greedy and BFS were both weak AI claims. Greedy just sorted. BFS just queried
  a hardcoded graph. No real optimization was happening.

What was built:
  1. src/composables/useDijkstra.js (new file)
     - Weighted shortest path through the pattern graph
     - Cost function: cost(pattern) = 4 − marksAtRisk
       High marks at risk → low cost → Dijkstra visits it first
     - Edge relaxation: if visiting pattern A reveals a cheaper path to neighbor B,
       update B's distance — this pulls connected high-risk patterns forward together
     - Output: ordered array of patterns = optimal study sequence
     - Time complexity: O(V²) with linear scan — fine for V ≤ 30

  2. src/views/StudyPathView.vue (new file)
     - New page at /study-path
     - Numbered list of all unmastered patterns in Dijkstra order
     - First pattern highlighted "Start here"
     - Mastered patterns shown with ✓ and greyed out
     - Each row links to that pattern's practice page

  3. src/router/index.js
     - Added route: /study-path → StudyPathView

  4. src/views/HomeView.vue
     - Added "View study path" link next to "View failure patterns" button

How Dijkstra differs from Greedy:
  Greedy: always picks the single highest-risk pattern, ignores connections
  Dijkstra: finds the minimum-cost PATH through the graph, so patterns
  connected to high-risk nodes get pulled forward as a cluster.
  Example: p012 (ln domain, medium risk) is connected to p007 (domain, high risk).
  Dijkstra may pull p012 forward after visiting p007 via edge relaxation.
  Greedy would leave p012 until all high-risk patterns are done.
